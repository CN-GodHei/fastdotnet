# Fastdotnet 插件系统设计与授权方案文档

## 1. 概述

本文档详细描述了 Fastdotnet 框架的插件系统架构设计，以及为保护开发者和运营者权益而制定的插件授权与验证方案。该方案旨在有效防止插件盗版，同时平衡开源项目的社区生态与商业化运营需求。

## 2. 插件系统架构

### 2.1 核心组件

Fastdotnet 插件系统由以下核心组件构成：

1.  **`Fastdotnet.Plugin.Contracts` (开源)**:
    *   定义了插件开发所需的所有公共接口和模型。
    *   包含 `IPlugin`, `IPermissionProvider`, `PluginConfig` 等核心契约。
    *   这是插件开发者唯一需要直接引用的公开库。

2.  **`Fastdotnet.Core` (开源)**:
    *   框架的核心库，包含基础模型、服务接口、工具类、常量等。
    *   除了授权验证相关的逻辑外，其余部分完全开源。

3.  **`Fastdotnet.Orm` (开源)**:
    *   框架的 ORM 实现库，基于 SqlSugar。
    *   完全开源。

4.  **`Fastdotnet.Plugin.Runtime` (闭源)**:
    *   插件系统的核心运行时库，包含插件加载、卸载、生命周期管理、授权验证等所有关键逻辑。
    *   **该库完全闭源**，是保护插件系统安全的核心。

### 2.2 依赖关系

```
Fastdotnet.WebApi (主程序)
    ├── Fastdotnet.Plugin.Runtime (私有 NuGet 包)
    │   ├── Fastdotnet.Plugin.Contracts (公开 NuGet 包/源代码)
    │   ├── Fastdotnet.Core (公开 NuGet 包/源代码)
    │   ├── Fastdotnet.Orm (公开 NuGet 包/源代码)
    │   ├── Autofac / Microsoft.AspNetCore.Mvc.Core / SqlSugarCore (公开 NuGet 包)
    │
    ├── Fastdotnet.Core (公开 NuGet 包/源代码)
    ├── Fastdotnet.Orm (公开 NuGet 包/源代码)
    └── 其他公开服务和模块...
```

主程序通过依赖注入与 `Fastdotnet.Plugin.Runtime` 提供的 `IPluginHost` 接口进行交互，对插件系统的具体实现细节无感知。

## 3. 插件授权与验证方案

为了防止插件盗版，我们采用基于部署数量的永久授权模式，并结合在线和离线验证机制。

### 3.1 授权模式

插件授权分为两种类型：

1.  **单机版 (Single Server License)**:
    *   授权仅限在一台物理或虚拟服务器上部署和运行。
    *   通过硬件指纹绑定来确保授权的唯一性。

2.  **多机版 (Multi-Server License)**:
    *   授权允许在指定数量的服务器上部署和运行。
    *   通过部署数量跟踪和硬件指纹登记来管理授权。

### 3.2 授权文件设计

授权文件采用 JSON 格式，并通过非对称加密进行数字签名以防止篡改。

```json
{
  "pluginId": "PluginA",
  "accountId": "purchaser@example.com", // 购买者账户ID
  "licenseType": "MultiServer", // or "SingleServer"
  "maxDeployments": 5, // 最大部署数量
  "issuedTo": "purchaser@example.com",
  "hardwareId": "ABC123XYZ", // 可选，用于单机版硬件绑定
  "signature": "..." // 使用私钥对以上内容进行签名
}
```

### 3.3 在线安装与验证

1.  **购买**: 用户在插件商城购买插件，选择授权类型。
2.  **安装**: 用户通过商城在线安装插件。
3.  **激活**:
    *   插件首次加载时，`Fastdotnet.Plugin.Runtime` 调用内部授权服务。
    *   服务向授权服务器发送请求，验证授权码和用户账户。
    *   验证通过后，服务器返回正式的、签名的授权文件并存储在插件目录。
4.  **定期验证**:
    *   插件运行期间，定期（如每天）通过后台任务向服务器确认授权状态。

### 3.4 离线安装与验证

#### 3.4.1 单机版离线授权

1.  **获取授权文件**: 用户在商城购买后，下载绑定到其账户的授权文件。
2.  **部署**: 用户将插件和授权文件部署到目标服务器。
3.  **激活**:
    *   插件首次运行时，读取授权文件，验证签名。
    *   收集服务器硬件信息并与授权文件中的 `hardwareId` 进行比对。
    *   如果匹配，则激活成功；否则，拒绝加载插件。

#### 3.4.2 多机版离线授权

1.  **获取授权文件**: 用户在商城购买后，下载包含 `maxDeployments` 数量的授权文件。
2.  **部署**: 用户将插件和授权文件部署到第一台服务器。
3.  **激活**:
    *   插件首次运行时，读取授权文件，验证签名。
    *   收集服务器硬件指纹 (HwF)。
    *   将 HwF 信息安全地记录在本地（如加密的登记文件）。
    *   更新本地记录，表示已使用1个授权。
4.  **部署到其他服务器**:
    *   用户将同一份插件和授权文件部署到其他服务器。
    *   新服务器上的插件激活时，会检查本地登记记录。
    *   如果已用授权数 < `maxDeployments` 且硬件指纹未登记，则登记并激活。
    *   否则，拒绝激活。

### 3.5 账户绑定与验证

为防止账户混用，授权验证流程中加入了账户匹配检查：

1.  **授权文件绑定**: 授权文件中明确记录购买者账户 ID (`accountId`)。
2.  **运行时账户上下文**: 框架提供 `IAccountContext` 服务，用于获取当前登录的账户 ID。
3.  **加载时验证**:
    *   插件加载时，`Fastdotnet.Plugin.Runtime` 不仅验证授权文件的有效性。
    *   还会通过 `IAccountContext` 获取当前账户 ID，并与授权文件中的 `accountId` 进行严格比对。
    *   只有账户匹配，插件才能被成功加载和启用。

## 4. 安全性与防护措施

### 4.1 代码层面

*   **核心逻辑闭源**: 最关键的加载和验证逻辑封装在 `Fastdotnet.Plugin.Runtime` 闭源库中。
*   **接口隔离**: 通过公开的 `IPluginHost` 接口与主程序交互，隐藏内部实现。
*   **代码混淆 (可选)**: 对 `Fastdotnet.Plugin.Runtime` 的 DLL 进行代码混淆，增加反编译难度。

### 4.2 授权层面

*   **数字签名**: 授权文件使用强非对称加密算法签名，防止伪造。
*   **硬件指纹**: 使用稳健的算法生成硬件指纹，适度容忍硬件变更。
*   **本地存储加密**: 对本地的授权登记记录进行加密存储。

## 5. 项目结构与分发

*   **开源仓库**: 包含 `Fastdotnet.Plugin.Contracts`, `Fastdotnet.Core`, `Fastdotnet.Orm` 以及其他公开模块的源代码。
*   **私有仓库/构建**: `Fastdotnet.Plugin.Runtime` 在独立的私有代码库中开发和维护。
*   **分发**:
    *   向社区发布开源框架，不包含插件运行时。
    *   向付费用户提供完整的框架，其中包含通过私有 NuGet 源或直接提供的 `Fastdotnet.Plugin.Runtime` 二进制文件。
