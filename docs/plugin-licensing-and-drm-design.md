# 插件授权与反盗版（DRM）方案设计文档

## 1. 目标

为了建立一个可持续的商业化插件生态系统，保护插件开发者和平台运营者的权益，我们需要设计一套安全、可靠且灵活的插件授权与验证机制，以有效防止盗版和未经授权的使用。

## 2. 核心原则

- **非对称加密**: 授权的核心是基于 RSA 非对称加密。授权服务器持有私钥用于签发授权，框架代码中只包含公钥用于验证，确保授权文件无法被伪造。
- **核心逻辑闭源**: 插件加载和授权验证的核心模块 `Fastdotnet.Plugin.Core` 必须闭源，以防止验证逻辑被轻易移除或篡改。项目将采用“开放核心 (Open-Core)”模式。
- **绑定硬件**: 对于单服务器授权，授权必须与服务器的唯一“机器指纹”绑定，防止授权文件被随意拷贝到其他服务器使用。

## 3. 核心概念

#### 3.1. 授权服务器 (License Server)

- **角色**: 插件商城的后端服务。
- **职责**:
    - 管理用户和插件购买记录。
    - 生成和签发授权文件 (`.lic`)。
    - 提供在线激活和离线激活支持。
    - 持有 **RSA 私钥**，用于对授权文件进行数字签名。

#### 3.2. 机器指纹 (Machine Fingerprint)

- **定义**: 一个能唯一标识一台服务器的字符串。
- **生成方式**: 通过采集服务器硬件信息（如 CPU ID, 主板序列号, 网卡 MAC 地址）组合并进行哈希计算得到。
- **作用**: 将“单服务器授权”与具体的物理或虚拟服务器进行强绑定。

#### 3.3. 授权文件 (`.lic`)

- **格式**: JSON 或 XML 文件，随插件一同分发。
- **核心字段**:
  ```json
  {
    "PluginId": "PluginA",
    "UserId": "user-guid-123",
    "LicenseType": "SingleServer", // SingleServer | MultiServer
    "MachineFingerprint": "hashed-fingerprint-string", // LicenseType为SingleServer时必需
    "IssueDate": "2025-08-31T00:00:00Z",
    "UpdatesUntil": "2026-08-31T00:00:00Z", // 订阅更新模式的截止日期
    "Signature": "rsa-sha256-signature-string" // 对以上所有字段内容的签名
  }
  ```
- **`Signature`**: 由授权服务器使用 **RSA 私钥**生成，是验证授权文件真实性的关键。

## 4. 商业模式

推荐采用 **“永久使用权 + 一年免费更新”** 的混合模式。

- **授权类型**:
    - `SingleServer`: 授权仅可用于一台服务器。
    - `MultiServer`: 授权可用于企业内部的多台服务器。
- **更新订阅**: 用户购买后可永久使用当前版本，但在一年内可免费获取所有新版本的更新。一年后，如需更新，则要续订。`UpdatesUntil` 字段用于控制此逻辑。

## 5. 实施策略与架构

### 5.1. “开放核心 (Open-Core)” 模式

- **开源部分**: 框架主体，如 `Fastdotnet.WebApi`, `Fastdotnet.Orm`, `Fastdotnet.Service` 以及 `Fastdotnet.Plugin.Contracts`。这部分保持开源以构建社区生态。
- **闭源部分**: `Fastdotnet.Plugin.Core`。此模块包含所有插件加载、管理和授权验证的逻辑。它将被开发于一个私有仓库，并作为编译后的 NuGet 包被主项目引用。

### 5.2. 验证流程

`PluginManager` 在加载每个插件时，必须执行以下验证：

1.  **查找 `.lic` 文件**: 检查插件目录下是否存在授权文件。
2.  **验证签名**: 使用预置在 `Fastdotnet.Plugin.Core` 中的 **RSA 公钥** 验证 `.lic` 文件的 `Signature` 是否合法。**失败则立即中止**，这能防止文件被篡改。
3.  **验证机器指纹**: 如果 `LicenseType` 为 `SingleServer`，则实时计算当前服务器的机器指纹，并与文件中的 `MachineFingerprint` 字段比对。**不匹配则中止**。
4.  **(可选) 验证更新资格**: 检查插件版本日期是否在 `UpdatesUntil` 日期之前。

## 6. 详细工作流程

### 6.1. 在线安装与激活

1.  **用户登录**: 管理员在后台登录插件商城账户。
2.  **请求安装**: 点击安装插件，Fastdotnet 后端向授权服务器发送请求，附带 `PluginId` 和当前服务器的**机器指纹**。
3.  **服务器签发**: 授权服务器验证购买记录，生成包含机器指纹和签名的 `.lic` 文件。
4.  **下载与安装**: 授权服务器将插件包和 `.lic` 文件一同返回，Fastdotnet 将其解压到正确的插件目录。
5.  **加载验证**: `PluginManager` 在加载插件时执行上述验证流程。

### 6.2. 离线安装与激活

1.  **生成请求**: 在无法联网的服务器上，Fastdotnet 生成一个包含**机器指纹**的请求文件 (`request.req`)。
2.  **上传请求**: 用户在插件商城网站上传 `request.req` 文件。
3.  **下载授权**: 授权服务器根据请求文件生成 `.lic` 文件，用户从网站下载插件包和 `.lic` 文件。
4.  **上传安装**: 用户回到 Fastdotnet 后台，上传插件包和 `.lic` 文件完成安装。

## 7. 安全性增强建议

- **原生代码模块**: 将最核心的逻辑（公钥存储、签名验证、机器指纹生成）使用 **C++ 或 Rust** 编写成原生库 (`.dll` 或 `.so`)，由 C# 通过 P/Invoke 调用。这能极大地提高逆向工程的难度。
- **代码混淆**: 对闭源的 `Fastdotnet.Plugin.Core.dll` 使用专业的 .NET 混淆工具（如 Obfuscar）进行处理，进一步加大反编译和分析的难度。

## 8. 执行步骤

### 第一阶段：授权服务器建设

1.  **技术选型**: 使用 ASP.NET Core Web API 构建授权服务器。
2.  **密钥生成**: 生成一对强 RSA 密钥（例如 4096 位），安全地存储私钥。
3.  **API 开发**:
    - 用户认证与管理 API。
    - 插件购买与授权记录 API。
    - 在线激活 API：接收机器指纹，返回插件包和 `.lic` 文件。
    - 离线激活 API：接收请求文件，返回 `.lic` 文件。

### 第二阶段：`Fastdotnet.Plugin.Core` 闭源与改造

1.  **创建私有仓库**: 将 `Fastdotnet.Plugin.Core` 项目迁移到一个新的私有 Git 仓库。
2.  **实现验证逻辑**:
    - 在项目中硬编码 **RSA 公钥**。
    - 实现机器指纹生成模块。
    - 在 `PluginLoadService` 或 `PluginManager` 中，添加完整的授权文件验证逻辑（签名、指纹等）。
3.  **打包发布**: 将该项目配置为自动打包成 NuGet 包。发布到一个私有的 NuGet 源或 `nuget.org`（作为非公开包）。
4.  **主项目迁移**:
    - 在 `Fastdotnet.sln` 中，移除对 `Fastdotnet.Plugin.Core` 的项目引用。
    - 添加对新发布的 NuGet 包的引用。

### 第三阶段：前端集成

1.  **开发管理界面**: 在 `fastdotnet-admin` 中开发插件商城相关的管理界面。
    - 登录/注销插件商城账户。
    - 已购插件列表、在线安装、卸载功能。
    - 离线激活功能（生成请求文件、上传授权文件）。

### 第四阶段：安全加固 (推荐)

1.  **开发原生库**: 使用 C++ 创建一个原生 DLL 项目。
2.  **迁移核心逻辑**: 将公钥存储、签名验证算法、机器指纹生成代码迁移到 C++ 库中。
3.  **集成**: 在 `Fastdotnet.Plugin.Core` 中使用 P/Invoke 调用此原生库。
4.  **代码混淆**: 在发布 `Fastdotnet.Plugin.Core` 的 NuGet 包之前，对其 `dll` 文件进行混淆处理。
