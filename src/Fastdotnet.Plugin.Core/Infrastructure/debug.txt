  概念详解：从一根火柴到一场大火

  1. 简单的火柴：Console.WriteLine
  在简单的控制台程序中，Console.WriteLine("Hello"); 就像划亮一根火柴，它直接、同步地在屏幕上打印出 "Hello"。火柴划完就灭了，不会留下任何后患。

  2. 智能的“纵火犯”：ASP.NET Core 日志框架
  在您的 ASP.NET Core Web API 项目中，环境完全不同。您看不到的 Program.cs 模板代码已经内置了一个强大的日志系统。这个系统会自动拦截所有 Console.WriteLine 的调用。

  当您的插件 PluginA 调用 Console.WriteLine("[PluginA] Initializing..."); 时，实际发生的事情是：

   * 步骤 A：拦截
      ASP.NET Core 的日志框架捕获了这个调用。它并没有立即把 "[PluginA] Initializing..." 这段文字直接扔到屏幕上。

   * 步骤 B：添加上下文（这是问题的根源！）
      为了打出您在日志中看到的、格式丰富的输出（比如时间戳、日志级别、来源类别），日志框架需要做额外的工作。它会检查：
       * “是谁在调用我？” -> “哦，是 PluginA.PluginAImpl 这个类。”
       * “这个类属于哪个程序集？” -> “它属于 PluginA.dll。”
       * 为了获取这些信息，日志框架必须加载并持有一个对 `PluginA.dll` 程序集的引用。

   * 步骤 C：异步处理
      为了不阻塞您的主程序，日志框架会将这个“待办事项”（即“打印这条来自 PluginA 的日志”）放入一个内部的队列中，然后让一个后台线程去慢慢处理。

  3. 致命的竞态条件：卸载程序 vs. 后台日志线程

  现在，最关键的场景出现了：

   1. 您的主线程执行 DisablePluginAsync，一路畅通无阻，最后调用了 alc.Unload()，命令操作系统：“卸载 PluginA.dll！”
   2. 几乎在同一时刻，日志框架的那个后台线程可能正好在处理来自 PluginA 的最后几条日志。
   3. 为了完成它的工作（打印出 info: PluginA.PluginAImpl[0] ...），这个后台线程依然持有对 `PluginA.dll` 的引用。
   4. 操作系统收到了卸载命令，但它检查后发现：“不行，还有个后台线程正在使用这个文件，我不能卸载。”
   5. 结果：文件被锁定。

   * 这个日志系统为了添加上下文，持有了对插件 DLL 的引用。
   * 由于日志处理是异步的，这个引用被一个后台线程持有，导致主线程的卸载命令无法完成。
